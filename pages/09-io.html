Input & Output

Reading input from the user and printing output — including fast I/O tricks critical for competitive programming.

📚 Simple Explanation

  • cin/cout: stream-based I/O from <iostream>.
  • << = insertion (output), >> = extraction (input).
  • printf/scanf: C-style; faster but error-prone.
  • Fast I/O trick: disable sync with stdio and untie cin from cout.

📄 Syntax Snapshot

#include <iostream>
using namespace std;

// Fast I/O (add at top of main)
ios_base::sync_with_stdio(false);
cin.tie(NULL);

int n;
cin >> n;                // read integer

string s;
getline(cin, s);         // read full line (with spaces)

cout << "Result: " << n << "\n";  // prefer \n over endl
// endl flushes the buffer — much slower!

▶ Example Code

#include <iostream>
#include <vector>
using namespace std;

int main() {
    ios_base::sync_with_stdio(false);
    cin.tie(NULL);

    int t;
    cin >> t;  // test cases
    while (t--) {
        int n;
        cin >> n;
        vector<int> a(n);
        for (int i = 0; i < n; i++) cin >> a[i];

        long long sum = 0;
        for (int x : a) sum += x;
        cout << sum << "\n";
    }
    return 0;
}

💡 Real-World Analogy

cin/cout are like a two-way intercom: >> listens to the incoming channel, << speaks into the outgoing. The sync-with-stdio trick is like switching from a slow shared radio to a private fast direct line.

⚠ Common Mistakes

  • Using endl in tight loops — it flushes the buffer and is 10-100x slower than "\n".
  • Mixing cin >> with getline without clearing the leftover newline.
  • Not enabling fast I/O for competitive problems — causes TLE on large inputs.

🌟 Interview Tips

  • Always add the two fast I/O lines in competitive coding solutions.
  • Know that scanf/printf are still valid and slightly faster than cin/cout even with sync off.
  • After cin >> n, add cin.ignore() before getline to consume the leftover newline.

📚 Simple Explanation

  • input(): reads one line as a string (always string — must cast).
  • print(): outputs to stdout with newline by default.
  • Fast I/O: sys.stdin.readline() is faster than input() for large data.
  • Use sys.stdout.write() instead of print() in tight output loops.

📄 Syntax Snapshot

name = input("Enter name: ")        # reads string
n = int(input())                    # cast to int
a, b = map(int, input().split())    # read two ints on one line
arr = list(map(int, input().split())) # read list

print("Hello", name, sep=", ", end="!\n")
print(f"Sum: {a + b}")              # f-string formatting

# Fast I/O for competitive programming
import sys
input = sys.stdin.readline          # override input
data = sys.stdin.read().split()     # read everything at once

▶ Example Code

import sys
input = sys.stdin.readline

def main():
    t = int(input())
    results = []
    for _ in range(t):
        n = int(input())
        arr = list(map(int, input().split()))
        results.append(str(sum(arr)))
    print("\n".join(results))  # single print call — faster

main()

💡 Real-World Analogy

input() is like a human operator taking one order at a time. sys.stdin.read().split() is like bulk-importing an entire order list from a file — a warehouse approach vs. retail counter.

⚠ Common Mistakes

  • Forgetting to cast: input() returns a string — int(input()) is required for math.
  • Using print() in a tight loop with millions of outputs — one large "\n".join() is far faster.
  • Calling input().split() when only one value is expected — safe, but int(input()) is cleaner.

🌟 Interview Tips

  • map(int, input().split()) is the canonical one-liner for reading multiple integers.
  • For competitive coding, always override input = sys.stdin.readline at the top.
  • Know that print is buffered — use print(..., flush=True) when interactive I/O is needed.

📚 Simple Explanation

  • Browser: No stdin; prompt with prompt(); output with console.log().
  • Node.js: Use readline module or process.stdin for console input.
  • For competitive coding, Node.js with readline is the standard setup.

📄 Syntax Snapshot

// Browser
const name = prompt("Your name?");
console.log("Hello", name);

// Node.js — readline approach
const readline = require("readline");
const rl = readline.createInterface({ input: process.stdin });
const lines = [];
rl.on("line", line => lines.push(line.trim()));
rl.on("close", () => {
  const n = parseInt(lines[0]);
  // process lines...
});

// Output
process.stdout.write("result\n");  // faster than console.log

▶ Example Code

// Competitive Node.js template
const readline = require("readline");
const rl = readline.createInterface({ input: process.stdin });
const lines = [];
rl.on("line", l => lines.push(l.trim()));
rl.on("close", () => {
    let idx = 0;
    const t = parseInt(lines[idx++]);
    const out = [];
    for (let i = 0; i < t; i++) {
        const n = parseInt(lines[idx++]);
        const arr = lines[idx++].split(" ").map(Number);
        out.push(arr.reduce((a, b) => a + b, 0));
    }
    console.log(out.join("\n"));
});

💡 Real-World Analogy

Node.js I/O is event-driven — like a restaurant where you place an order (rl.on("line")), and the chef (event loop) alerts you when food is ready ("close"). You can't demand food the instant you order.

⚠ Common Mistakes

  • Trying to use prompt() in Node.js — it doesn't exist there.
  • Reading data before the "close" event fires — async nature bites beginners.
  • Using console.log in a tight loop — process.stdout.write is faster.

🌟 Interview Tips

  • For online judge problems in JS, always collect all lines first in "line" events, process in "close".
  • console.log adds a newline; process.stdout.write does not.
  • In browser context, console.log accepts multiple args and objects — useful for debugging.
