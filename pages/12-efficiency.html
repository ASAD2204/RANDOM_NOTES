Code Efficiency

Big-O complexity, space-time trade-offs, and optimization patterns that separate good from great solutions.

📚 Simple Explanation

Big-O quick reference:

ComplexityNameExample
O(1)ConstantArray index, hash lookup
O(log N)LogarithmicBinary search, balanced BST
O(N)LinearSingle loop, linear scan
O(N log N)Linearithmicstd::sort, merge sort
O(N²)QuadraticNested loop, bubble sort
O(2^N)ExponentialBrute-force subsets

📄 Syntax Snapshot

// O(N²) naive — check all pairs
for (int i = 0; i < n; i++)
    for (int j = i+1; j < n; j++)
        if (a[i]+a[j] == target) return {i, j};

// O(N) optimized — hash map
unordered_map<int,int> seen;
for (int i = 0; i < n; i++) {
    int complement = target - a[i];
    if (seen.count(complement)) return {seen[complement], i};
    seen[a[i]] = i;
}

▶ Example Code

#include <unordered_map>
#include <vector>
using namespace std;

// Two Sum: O(N) time, O(N) space
vector<int> twoSum(vector<int>& nums, int target) {
    unordered_map<int,int> seen;  // value → index
    for (int i = 0; i < (int)nums.size(); i++) {
        int comp = target - nums[i];
        if (seen.count(comp)) return {seen[comp], i};
        seen[nums[i]] = i;
    }
    return {};
}

// Reserve capacity to avoid reallocations
vector<int> v;
v.reserve(1000);  // pre-allocate
for (int i = 0; i < 1000; i++) v.push_back(i);

💡 Real-World Analogy

Optimizing code is like finding a person in a city. Checking every house (O(N)) works but is slow. Using a directory/index (O(log N)) is faster. Knowing their GPS coordinates (O(1) hash lookup) is instant — the hash map is that directory.

⚠ Common Mistakes

  • Nested loops when a hash map solution exists — automatic O(N²) when O(N) is possible.
  • Not considering space complexity — sometimes O(N) extra space is the trade-off.
  • Premature vector reallocation — use reserve() when size is known.

🌟 Interview Tips

  • Always state your time and space complexity before coding — it shows the interviewer you think before you act.
  • Common trick: sort first, then binary search or two-pointer → reduces O(N²) to O(N log N).
  • Hash maps (unordered_map) trade O(N) space for O(1) lookup — explain this trade-off explicitly.

📄 Syntax Snapshot

# O(N²) naive
def two_sum_naive(nums, target):
    for i in range(len(nums)):
        for j in range(i+1, len(nums)):
            if nums[i] + nums[j] == target:
                return [i, j]

# O(N) optimized
def two_sum(nums, target):
    seen = {}
    for i, n in enumerate(nums):
        comp = target - n
        if comp in seen:
            return [seen[comp], i]
        seen[n] = i

▶ Example Code

# Sliding window — O(N) for max subarray of size k
def max_window_sum(nums, k):
    window = sum(nums[:k])
    best = window
    for i in range(k, len(nums)):
        window += nums[i] - nums[i - k]
        best = max(best, window)
    return best

# Memoization for O(N) fibonacci instead of O(2^N)
from functools import lru_cache

@lru_cache(maxsize=None)
def fib(n):
    if n <= 1: return n
    return fib(n-1) + fib(n-2)

print(fib(100))  # instant

💡 Real-World Analogy

Memoization is like a student writing down answers to previously solved problems — instead of recalculating every time, they flip back to their notes. @lru_cache is Python's built-in "answer notebook".

⚠ Common Mistakes

  • Using list where deque is needed — list.pop(0) is O(N) every time.
  • String concatenation in a loop: O(N²); use "".join(parts).
  • Recursive fibonacci without memoization — exponential time on large N.

🌟 Interview Tips

  • Sliding window, two-pointer, and hash map are the three O(N) patterns that solve 80% of medium problems.
  • @lru_cache on a recursive function instantly applies top-down DP.
  • Know when to prefer bottom-up DP over recursion (avoids Python's recursion limit of ~1000).

📄 Syntax Snapshot

// O(N²) naive
function twoSumNaive(nums, target) {
    for (let i = 0; i < nums.length; i++)
        for (let j = i+1; j < nums.length; j++)
            if (nums[i] + nums[j] === target) return [i, j];
}

// O(N) optimized
function twoSum(nums, target) {
    const seen = new Map();
    for (let i = 0; i < nums.length; i++) {
        const comp = target - nums[i];
        if (seen.has(comp)) return [seen.get(comp), i];
        seen.set(nums[i], i);
    }
}

▶ Example Code

// Memoization with a Map (manual)
function memoize(fn) {
    const cache = new Map();
    return function(...args) {
        const key = JSON.stringify(args);
        if (cache.has(key)) return cache.get(key);
        const result = fn.apply(this, args);
        cache.set(key, result);
        return result;
    };
}

const fib = memoize(function(n) {
    if (n <= 1) return n;
    return fib(n-1) + fib(n-2);
});

console.log(fib(50)); // instant

💡 Real-World Analogy

A Map lookup is like having a perfectly indexed library catalog — find any book in O(1). A linear array includes() is like reading every title on every shelf until you find the one you want — O(N).

⚠ Common Mistakes

  • Using arr.includes(x) inside a loop when a Set lookup would be O(1).
  • Not providing a sort comparator — lexicographic default causes subtle bugs.
  • Excessive DOM manipulation in a loop — batch changes, or use DocumentFragment.

🌟 Interview Tips

  • Whenever you hear "find if exists" in an O(N) context — think Set.
  • Whenever you hear "count occurrences" — think Map.
  • Sliding window is just as applicable in JS — window sum/product patterns come up in frontend data processing too.
