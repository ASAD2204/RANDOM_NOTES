Error Handling

Exceptions, try-catch-finally, custom errors, and graceful failure strategies across all three languages.

📚 Simple Explanation

  • throw raises an exception; try/catch handles it.
  • Can throw any type — best practice: throw objects derived from std::exception.
  • catch(...) is a catch-all (avoid in production).
  • C++ does not have a finally block — use destructors (RAII) instead.

📄 Syntax Snapshot

#include <stdexcept>
#include <iostream>

try {
    if (x == 0) throw std::runtime_error("Division by zero");
    int result = 10 / x;
}
catch (const std::runtime_error& e) {
    std::cerr << "Error: " << e.what() << "\n";
}
catch (const std::exception& e) {
    std::cerr << "Exception: " << e.what() << "\n";
}
catch (...) {
    std::cerr << "Unknown error\n";
}

▶ Example Code

#include <stdexcept>
#include <iostream>

// Custom exception
class ValidationError : public std::runtime_error {
public:
    explicit ValidationError(const std::string& msg)
        : std::runtime_error(msg) {}
};

int divide(int a, int b) {
    if (b == 0) throw ValidationError("Denominator cannot be zero");
    return a / b;
}

int main() {
    try {
        std::cout << divide(10, 0) << "\n";
    } catch (const ValidationError& e) {
        std::cerr << "Validation: " << e.what() << "\n";
    }
    return 0;
}

💡 Real-World Analogy

try is like attempting to defuse a bomb. throw is the bomb going off. catch is the safety protocol handler. In C++, RAII (destructors) acts as the automatic cleanup crew that always runs, even when things explode.

⚠ Common Mistakes

  • Catching by value instead of const reference — slices the exception type.
  • Throwing from a destructor — causes immediate program termination.
  • Using exceptions for normal control flow — bad for performance and readability.
  • Catching std::exception& before a more specific type — the specific catch is never reached.

🌟 Interview Tips

  • Explain RAII as C++'s replacement for finally — automatically run destructors guarantee cleanup.
  • Know the noexcept specifier — marks functions that promise not to throw.
  • Interviewers like: "What happens if an exception is thrown in a constructor?"

📚 Simple Explanation

  • try/except/else/finally: full exception lifecycle.
  • else block runs only if no exception was raised — unique to Python!
  • finally always runs — used for cleanup (close files, release locks).
  • Built-in hierarchy: BaseException → Exception → specific errors.

📄 Syntax Snapshot

try:
    result = 10 / x
except ZeroDivisionError:
    print("Can't divide by zero")
except (TypeError, ValueError) as e:
    print(f"Type/Value error: {e}")
except Exception as e:
    print(f"Unexpected: {e}")
    raise  # re-raise
else:
    print(f"Success: {result}")  # runs only if no exception
finally:
    print("This always runs")    # cleanup

▶ Example Code

# Custom exception
class InsufficientFundsError(Exception):
    def __init__(self, amount, balance):
        super().__init__(f"Need {amount}, have {balance}")
        self.amount = amount
        self.balance = balance

def withdraw(amount, balance):
    if amount > balance:
        raise InsufficientFundsError(amount, balance)
    return balance - amount

try:
    new_balance = withdraw(100, 50)
except InsufficientFundsError as e:
    print(f"Error: {e}")  # Error: Need 100, have 50
finally:
    print("Transaction attempted")  # always prints

💡 Real-World Analogy

try is the risky experiment. except is the safety net. else is the celebration party when nothing went wrong. finally is the janitor who cleans the lab regardless of outcome.

⚠ Common Mistakes

  • Catching bare Exception silently: except: pass hides all bugs — always handle or re-raise.
  • Catching Exception instead of BaseException — misses KeyboardInterrupt, SystemExit.
  • Forgetting to raise after logging — error gets swallowed.

🌟 Interview Tips

  • The else clause is Python-specific — mention it in interviews to stand out.
  • Context managers (with open(...)) use __enter__/__exit__ — cleaner than finally for resource management.
  • Know common built-in exceptions: ValueError, TypeError, KeyError, IndexError, AttributeError.

📚 Simple Explanation

  • try/catch/finally: synchronous error handling.
  • For async code: .catch() on Promises or try/catch with async/await.
  • throw can throw anything — best practice: throw Error objects (have a stack trace).

📄 Syntax Snapshot

// Synchronous
try {
    if (x === 0) throw new Error("Division by zero");
    const result = 10 / x;
} catch (e) {
    console.error(e.message);
} finally {
    console.log("Always runs");
}

// Async/Await
async function fetchData(url) {
    try {
        const response = await fetch(url);
        if (!response.ok) throw new Error(`HTTP ${response.status}`);
        return await response.json();
    } catch (e) {
        console.error("Fetch failed:", e.message);
        throw e;  // re-throw so caller can handle
    }
}

▶ Example Code

// Custom error class
class ValidationError extends Error {
    constructor(field, message) {
        super(message);
        this.name = "ValidationError";
        this.field = field;
    }
}

function validateAge(age) {
    if (typeof age !== "number") throw new TypeError("Age must be a number");
    if (age < 0 || age > 150) throw new ValidationError("age", "Out of valid range");
    return true;
}

try {
    validateAge(-5);
} catch (e) {
    if (e instanceof ValidationError) {
        console.log(`Field '${e.field}': ${e.message}`);
    } else {
        throw e;  // unexpected error — re-throw
    }
}

💡 Real-World Analogy

Synchronous errors are like tripping while walking — caught immediately. Async errors with Promises are like ordering delivery — if something goes wrong, you only find out when the delivery fails. async/await makes async look synchronous, like checking a tracking app that appears live.

⚠ Common Mistakes

  • Forgetting await in an async function — unhandled promise rejection.
  • Throwing a non-Error value: throw "message" — no stack trace.
  • Empty catch blocks that swallow errors silently.
  • Using try/catch inside a .then() — just use .catch() on the chain instead.

🌟 Interview Tips

  • Always explain unhandled promise rejections — common interview topic in Node.js.
  • Know how to build a custom error class extending Error (set this.name).
  • Promise.allSettled vs Promise.all: former continues even if one rejects.
