Interview Q&A

Exam-ready model answers for the most commonly asked language theory questions in placement interviews.

📚 Simple Explanation

C++ interview questions focus on: pointers vs references, memory model, OOP fundamentals, templates, and STL internals.

📄 Q&A Quick Reference

Q: Pointer vs Reference?
A: Pointer — can be null, reassigned, needs *.
   Reference — must be initialized, can't be null, no *.

Q: Stack vs Heap?
A: Stack — fast, auto-freed, limited size (usually 1-8 MB).
   Heap — slow, manual (or smart ptr), virtually unlimited.

Q: What is RAII?
A: Resource acquisition in constructor, release in destructor.
   Ensures cleanup even when exceptions occur.

Q: What is const-correctness?
A: Using const to prevent unintended mutation.
   const ref parameters avoid copies AND mutation.

▶ Top 10 C++ Interview Questions

// Q1: What does 'virtual' do?
// A: Enables runtime polymorphism via vtable.
class Base { virtual void f() {} };
// Without virtual: Base::f called even with Derived pointer.
// With virtual: Derived::f called (dynamic dispatch).

// Q2: What is undefined behavior (UB)?
// A: Code the standard doesn't define behavior for.
// Examples: buffer overflow, null dereference, signed int overflow.

// Q3: When to use struct vs class?
// A: struct members public by default; class members private.
// Convention: struct for POD data, class for OOP entities.

// Q4: What is a copy constructor?
class Widget {
    Widget(const Widget& other); // copies another Widget
};

// Q5: What does 'explicit' keyword do?
struct Kg { explicit Kg(double v) : val(v) {} double val; };
// Prevents: Kg k = 5.0; (implicit conversion)
// Allows:  Kg k(5.0);   (explicit)

💡 Real-World Analogy

A virtual function table (vtable) is like a company directory: instead of going directly to a person, you look up their current role in the directory — the same role title may be filled by different people depending on the company branch (derived class).

⚠ Common Mistakes

  • Not making destructors virtual in base classes — derived destructor never called, causing memory leak.
  • Confusing const int* (pointer to const) with int* const (const pointer).
  • Slicing: assigning a derived object to a base object by value loses the derived part.

🌟 Interview Tips

  • OOP concepts to memorize: encapsulation, inheritance, polymorphism, abstraction.
  • Rule of three/five: if you define one of destructor/copy ctor/copy assign — define all of them.
  • nullptr vs NULL: nullptr is type-safe (C++11+), always prefer it.

📚 Simple Explanation

Python interview questions center on: mutability, pass semantics, GIL, decorators, generators, and OOP dunder methods.

📄 Q&A Quick Reference

Q: Is Python pass-by-value or pass-by-reference?
A: Neither — pass-by-object-reference (pass-by-assignment).
   Immutable objects behave like value; mutable like reference.

Q: What is the GIL?
A: Global Interpreter Lock — only one thread runs Python bytecode
   at a time. Use multiprocessing for CPU parallelism.

Q: What is a generator?
A: Function using 'yield'; produces values lazily on demand.
   Memory efficient for large sequences.

Q: What is a decorator?
A: A function that wraps another function to add behavior.
   @decorator == func = decorator(func)

▶ Top 10 Python Interview Questions

# Q1: What are *args and **kwargs?
def f(*args, **kwargs):
    print(args)   # tuple of positional args
    print(kwargs) # dict of keyword args

# Q2: Difference between list, tuple, set, dict?
# list: ordered, mutable, allows duplicates
# tuple: ordered, immutable, allows duplicates
# set: unordered, mutable, unique elements
# dict: key-value, ordered (Python 3.7+)

# Q3: What is __init__ vs __new__?
# __new__ creates the instance; __init__ initializes it.

# Q4: How does Python manage memory?
# Ref-counting + cyclic garbage collector.
# Objects freed when ref count reaches 0.

# Q5: What is a class method vs static method?
class C:
    @classmethod
    def cm(cls): pass    # receives class as first arg

    @staticmethod
    def sm(): pass        # no implicit first arg

💡 Real-World Analogy

Python's pass-by-object-reference is like giving someone the address of a house. If they redecorate inside (mutate), the change is visible to all. But if they move to a different house (reassign), only their copy of the address changes — the original stays.

⚠ Common Mistakes

  • Confusing is vs == — always use == for value equality.
  • Forgetting self in instance methods — TypeError: takes 0 positional arguments.
  • Using threads for CPU-bound work when GIL is a bottleneck — use multiprocessing instead.

🌟 Interview Tips

  • Dunder (magic) methods to know: __init__, __str__, __repr__, __len__, __eq__, __hash__.
  • Generator expression vs list comprehension: generators are lazy (memory-efficient).
  • Decorators: know how to write a timing decorator with functools.wraps.

📚 Simple Explanation

JS interviews love: closures, event loop, scope (var/let/const), prototypal inheritance, Promises, and this binding.

📄 Q&A Quick Reference

Q: == vs === ?
A: == coerces types before comparison; === does not.
   Always use === unless you intentionally want coercion.

Q: What is a closure?
A: A function that "closes over" variables from its outer scope.
   The inner function remembers those variables even after
   the outer function has returned.

Q: What is event loop?
A: JS runs code, collects & processes events, executes
   queued sub-tasks. Call stack → microtask queue (Promises)
   → macrotask queue (setTimeout).

Q: var vs let vs const?
A: var — function-scoped, hoisted (initialized as undefined).
   let — block-scoped, hoisted but NOT initialized (TDZ).
   const — block-scoped, must be assigned at declaration.

▶ Top 10 JS Interview Questions

// Q1: What is hoisting?
console.log(x); // undefined (var hoisted)
var x = 5;
console.log(y); // ReferenceError (let in TDZ)
let y = 5;

// Q2: Explain closure
function counter() {
    let count = 0;
    return () => ++count; // closes over 'count'
}
const c = counter();
c(); // 1
c(); // 2 — count persists

// Q3: What does 'this' refer to in arrow vs regular function?
const obj = {
    name: "Alice",
    greet: function() { console.log(this.name); },  // "Alice"
    wave: () => { console.log(this.name); }         // undefined (lexical this)
};

// Q4: Explain Promise states
// Pending → Fulfilled (resolve) or Rejected (reject)
// .then() handles fulfilled, .catch() handles rejected

// Q5: What is the prototype chain?
function Animal(name) { this.name = name; }
Animal.prototype.speak = function() { return this.name + " speaks"; };
const dog = new Animal("Rex");
dog.speak(); // Rex speaks — found via prototype

💡 Real-World Analogy

The JS event loop is like a single chef in a kitchen: works on one task at a time (call stack), but takes notes (async queue) when meals need more time. When the current dish is done, checks the notes and works on the next pending item — never truly parallel, but never idle.

⚠ Common Mistakes

  • Losing this context — a method passed as a callback loses its object binding.
  • Temporal Dead Zone (TDZ): accessing let/const before declaration throws ReferenceError.
  • Confusing microtasks (Promise callbacks) and macrotasks (setTimeout) order in event loop.

🌟 Interview Tips

  • Closure is possibly the #1 JS interview topic — write one from scratch without prompting.
  • Event loop question: console.log(1); setTimeout(f, 0); console.log(2); — what order? 1, 2, f.
  • Know call, apply, bind: all three manually set this.
