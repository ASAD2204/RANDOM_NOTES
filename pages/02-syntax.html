Syntax Fundamentals

How to structure a program: blocks, statements, comments, entry points, and whitespace rules.

📚 Simple Explanation

C++ has strict, compiler-enforced syntax. Key rules:

  • Every statement ends with a semicolon ;.
  • Code blocks are wrapped in curly braces { }.
  • Program starts at int main().
  • Headers are included with #include <header>.
  • Case-sensitive: Main ≠ main.

📄 Syntax Snapshot

#include <library>   // include headers

// Single-line comment
/* Multi-line
   comment */

int main() {
    // code here
    return 0;       // mandatory
}

▶ Example Code

#include <iostream>

int main() {
    // Declare and print
    int age = 20;
    std::cout << "Age: " << age << "\n";

    if (age >= 18) {
        std::cout << "Adult\n";
    }
    return 0;
}

💡 Real-World Analogy

Like a legal contract: every clause must end with a period, every section in brackets, every paragraph labelled — miss one rule and the contract is void (code won't compile).

⚠ Common Mistakes

  • Missing ; at end of statements (especially after class/struct closing }).
  • Mismatched braces {}.
  • Forgetting return 0; in main (technically UB before C++11).
  • Using // comment on a preprocessor line like #include.

🌟 Interview Tips

  • Always write clean, compilable code in interviews — missing semicolons are red flags.
  • Use std:: prefix or a concise using namespace std; (explain the tradeoff).
  • Consistent indentation signals professionalism even more than correctness.

📚 Simple Explanation

Python uses indentation (not braces) to define code blocks. Key rules:

  • No semicolons required — newline ends a statement.
  • Indentation is mandatory — typically 4 spaces.
  • Block-opening lines end with a colon :.
  • No explicit entry point — code runs top-to-bottom.

📄 Syntax Snapshot

# Single-line comment

"""
Multi-line string (docstring)
used as block comments
"""

def function_name():
    # 4-space indent = block
    pass

if __name__ == "__main__":
    function_name()

▶ Example Code

age = 20      # no type declaration needed

if age >= 18:
    print("Adult")   # indent defines the block
else:
    print("Minor")

💡 Real-World Analogy

Like a well-formatted book: paragraphs (blocks) are identified by their indentation on the page. No need for quotation marks or brackets to show where one paragraph ends.

⚠ Common Mistakes

  • Mixing tabs and spaces — Python raises TabError.
  • Forgetting the colon after if, for, def.
  • Incorrect indentation level breaks logic silently (code runs but does wrong thing).
  • Expecting global scope to act like local scope for mutable variables.

🌟 Interview Tips

  • Always use 4 spaces; never tabs in interview code.
  • Python's readable syntax is a strength — keep code looking like clean English.
  • If asked "why no braces?": "Indentation enforces readability by design (Guido van Rossum's philosophy)."

📚 Simple Explanation

JavaScript syntax is C-family: curly braces define blocks, semicolons end statements. However, ASI (Automatic Semicolon Insertion) means semicolons are technically optional — but relying on it is risky.

  • Blocks use { }.
  • Semicolons recommended (use a linter like ESLint).
  • Case-sensitive.
  • Can run without a main() function.

📄 Syntax Snapshot

// Single-line comment
/* Multi-line comment */

function greet(name) {
  // body
  return "Hello " + name;
}

greet("World");

▶ Example Code

const age = 20;

if (age >= 18) {
  console.log("Adult");
} else {
  console.log("Minor");
}

💡 Real-World Analogy

Like informal English writing: you can skip periods (semicolons) and people may still understand you — but sometimes the meaning changes unexpectedly. Better to be explicit.

⚠ Common Mistakes

  • Relying on ASI with return on its own line — JS inserts a semicolon after return and returns undefined.
  • Forgetting to handle the async nature — synchronous mental model applied to async code.
  • Global variable pollution by forgetting let/const/var.

🌟 Interview Tips

  • Always use const by default; switch to let only when the value changes.
  • Use a consistent style — interviewers watch for structured thinking.
  • Mention "use strict"; — it prevents many silent bugs and is good practice.
