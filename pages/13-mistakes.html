Common Mistakes

The most frequent errors beginners — and even experienced developers — make. Study these to avoid losing marks.

📚 Simple Explanation

C++ mistakes are especially punishing because many cause undefined behavior (UB) — the program may silently produce wrong results, crash, or work sometimes but not others.

📄 Mistake Patterns

// ❌ Integer overflow — silent!
int a = 2000000000;
int b = a + a;  // overflows int range

// ✅ Use long long
long long b = (long long)a + a;

// ❌ Comparing size_t (unsigned) with int
vector<int> v = {1,2,3};
for (int i = v.size()-1; i >= 0; i--) { /* ... */ }
// When v is empty: v.size()-1 wraps around to HUGE number!

// ✅ Cast explicitly
for (int i = (int)v.size()-1; i >= 0; i--) { /* ... */ }

▶ Top 5 Mistakes with Fixes

// 1. Off-by-one in loops
for (int i = 0; i <= n; i++)   // ❌ i<=n goes out of bounds
for (int i = 0; i < n; i++)    // ✅

// 2. Using [] on map creates default entry
map<string,int> m;
if (m["key"] == 0) // ❌ creates "key" with value 0!
if (m.count("key") && m["key"] == 0) // ✅

// 3. Dangling reference
string& getName() {
    string s = "local";
    return s;  // ❌ returns ref to destroyed local variable!
}

// 4. Forgetting to handle empty input
vector<int> v;
int mx = v[0]; // ❌ UB on empty vector
int mx = v.empty() ? INT_MIN : v[0]; // ✅

// 5. endl vs \n in loops
for (int i=0; i<1000000; i++)
    cout << i << endl;  // ❌ 1M flushes — very slow
    cout << i << "\n";  // ✅

💡 Real-World Analogy

C++ is like driving a race car — fast and powerful, but no guardrails. One wrong turn (UB) and the car may go anywhere. The compiler doesn't always warn you. Learn the dangerous roads by heart.

⚠ Additional Pitfalls

  • Division of negative integers: -7 / 2 = -3 in C++ (truncates toward zero).
  • Modulo with negatives: -7 % 3 = -1 in C++ — can break index calculations.
  • Forgetting #include — different compilers behave differently for missing headers.

🌟 Interview Tips

  • When asked "can X go wrong?" — think overflow, empty container, dangling pointer, UB.
  • Always use long long for sums of large arrays in competitive coding.
  • Use assert() during development to catch logic bugs early.

📚 Simple Explanation

Python mistakes are often silent logic errors or type surprises — the code runs but produces wrong output. Python's flexibility is also its trickiness.

📄 Mistake Patterns

# ❌ Mutable default argument — shared across all calls!
def append_to(item, lst=[]):
    lst.append(item)
    return lst

append_to(1)  # [1]
append_to(2)  # [1, 2] — not [2]!

# ✅ Use None
def append_to(item, lst=None):
    if lst is None: lst = []
    lst.append(item)
    return lst

# ❌ Late binding closures in loops
fns = [lambda: i for i in range(3)]
print(fns[0]())  # 2 — not 0!

# ✅ Capture with default arg
fns = [lambda i=i: i for i in range(3)]

▶ Top 5 Mistakes with Fixes

# 1. == vs is for comparison
x = [1,2,3]
y = [1,2,3]
x == y  # True (same value)
x is y  # False (different objects)
# Use == for values, is for None/True/False checks only

# 2. Forgetting integer division
7 / 2    # 3.5 (float in Python 3)
7 // 2   # 3   (integer floor division)

# 3. Modifying list while iterating
lst = [1,2,3,4]
for x in lst:
    if x % 2 == 0: lst.remove(x)  # ❌ skips elements
lst = [x for x in lst if x % 2 != 0]  # ✅

# 4. String immutability
s = "hello"
s[0] = "H"  # ❌ TypeError
s = "H" + s[1:]  # ✅

# 5. Chained comparison (works in Python, looks wrong)
1 < x < 10  # ✅ valid Python — both conditions checked

💡 Real-World Analogy

Python's mutable default argument is like a shared notepad in an office — everyone thinks they have their own but they're all writing and reading from the same one. The fix is giving each caller a fresh notepad (None sentinel).

⚠ Additional Pitfalls

  • dict.keys() returns a view, not a list — mutate the dict and the view changes.
  • Global variable in a function: reading is fine, but assigning creates a local — use global keyword.
  • Unpacking mismatch: a, b = [1, 2, 3] raises ValueError.

🌟 Interview Tips

  • The mutable default argument trap is a classic Python interview question — know it cold.
  • Late-binding closures in loops is another classic — be ready with the i=i fix.
  • Python integer division: always clarify // vs / when writing solutions.

📚 Simple Explanation

JavaScript has many type coercion surprises and scoping gotchas that trip up developers at every level. Many stem from legacy design decisions kept for backward compatibility.

📄 Mistake Patterns

// ❌ == coerces types — surprises everywhere
"" == false     // true
0 == false      // true
null == undefined // true
[] == false     // true!

// ✅ Always use ===
"" === false    // false
0 === false     // false

// ❌ var in loops creates function scope, not block scope
for (var i = 0; i < 3; i++) {
    setTimeout(() => console.log(i), 100);  // logs 3, 3, 3
}
// ✅ Use let for block scope
for (let i = 0; i < 3; i++) {
    setTimeout(() => console.log(i), 100);  // logs 0, 1, 2
}

▶ Top 5 Mistakes with Fixes

// 1. Array sort default is lexicographic
[10, 9, 2].sort()             // ❌ [10, 2, 9]
[10, 9, 2].sort((a,b) => a-b) // ✅ [2, 9, 10]

// 2. typeof null === "object" (historic bug)
typeof null   // "object" — NOT "null"!
null === null // ✅ use strict equality

// 3. this context lost in callbacks
class Timer {
    start() { setTimeout(function() { console.log(this); }, 100); }
    // this is undefined in strict mode!
    startFixed() { setTimeout(() => console.log(this), 100); } // ✅ arrow fn
}

// 4. NaN comparisons
NaN === NaN  // false — NaN is not equal to itself!
Number.isNaN(NaN)  // ✅ true

// 5. Floating point arithmetic
0.1 + 0.2 === 0.3  // false!
Math.abs(0.1 + 0.2 - 0.3) < Number.EPSILON  // ✅

💡 Real-World Analogy

JS type coercion is like a very accommodating waiter who guesses what you meant instead of asking — sometimes brilliantly helpful, sometimes horrifyingly wrong. === is asking the waiter to take your order literally.

⚠ Additional Pitfalls

  • + operator with strings vs numbers: "3" + 3 = "33" but "3" - 3 = 0.
  • Async functions always return a Promise — forgetting await gives you a Promise object, not the value.
  • Object mutation through passed references — accidentally modifying inputs is a common bug.

🌟 Interview Tips

  • var vs let in loops and closures is an evergreen JS interview question.
  • Always explain why === is preferred over == — not just "use strict equality".
  • NaN !== NaN is a gotcha every JS interviewer expects you to know.
