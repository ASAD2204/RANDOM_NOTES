Hash Tables

Hash functions, collisions, chaining, and open addressing in interview context.

📚 Concept

Hash table maps key → index using hash function. Average lookup/insert/delete O(1).

💡 Analogy

Library indexing: a code maps directly to shelf bucket; collisions mean multiple books in same slot.

📄 Step-by-step (insert with chaining)

1) idx = hash(key) % capacity
2) Go to bucket idx
3) If key exists update value
4) Else append new key-value pair to bucket list

▶ Implementations

// C++
unordered_map mp; mp["a"]++;

# Python
d={}; d["a"]=d.get("a",0)+1

// JS
const mp=new Map(); mp.set("a", (mp.get("a")||0)+1);

⏱ Complexity

  • Average search/insert/delete: O(1)
  • Worst-case (many collisions): O(n)
  • Space: O(n)

✅ / ❌

  • ✅ Very fast average-case operations.
  • ❌ Unordered, collision overhead, worst-case degradation.

⚠ Common Mistakes

  • Assuming deterministic O(1) in all cases.
  • Bad hash function causing cluster collisions.
  • Forgetting rehash/load factor effects.

🌟 Interview Tips

  • Hash map is default for frequency counting, two-sum, grouping.
  • Mention collision strategy when interviewer asks internals.
