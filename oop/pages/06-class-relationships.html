Relationships Between Classes

Association, Aggregation, and Composition are foundational UML and interview concepts.

📄 Diagram

Association:  Teacher -------- Student   (knows/uses)
Aggregation:  Team <>-------- Player     (whole-part, weak ownership)
Composition:  House *--------- Room       (whole-part, strong ownership)

📚 Definitions

  • Association: general relationship; lifecycles independent.
  • Aggregation: “has-a” with weak ownership; part can outlive whole.
  • Composition: strong ownership; part lifecycle tied to whole.

▶ Example Pattern

class Player {};
class Team { vector<Player*> players; }; // aggregation (external players)

class Room {};
class House { vector<Room> rooms; };     // composition (rooms owned by house)

🌟 Interview Insight

When interviewer asks “difference between aggregation and composition,” answer using ownership and lifecycle, not just “both are has-a”.

▶ Theory Deep Dive

Object-oriented design should be explained through abstraction boundaries, coupling/cohesion, and maintainability trade-offs.
