SOLID Principles

Five design principles for maintainable, testable object-oriented systems.

📚 S — Single Responsibility Principle

A class should have one reason to change. Split unrelated concerns (e.g., InvoiceCalculator vs InvoicePrinter).

📚 O — Open/Closed Principle

Software entities should be open for extension, closed for modification. Add new behavior through subclasses/strategies, not by editing stable core logic.

📚 L — Liskov Substitution Principle

Subtypes must be replaceable for base types without breaking correctness. If subclass weakens postconditions or breaks contract, it violates LSP.

📚 I — Interface Segregation Principle

Clients should not depend on methods they do not use. Prefer small focused interfaces over one fat interface.

📚 D — Dependency Inversion Principle

High-level modules should depend on abstractions, not concretions. Inject dependencies (constructor injection) to improve testability.

▶ Tiny Example (DIP + OCP)

class Notifier { public: virtual void send(string)=0; virtual ~Notifier()=default; };
class EmailNotifier : public Notifier { public: void send(string m) override {/*email*/} };
class SmsNotifier   : public Notifier { public: void send(string m) override {/*sms*/} };

class AlertService {
  Notifier& notifier;
public:
  AlertService(Notifier& n): notifier(n) {}
  void trigger(){ notifier.send("ALERT"); }
};
