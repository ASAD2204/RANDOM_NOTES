Introduction to Object-Oriented Programming

What OOP is, why it matters, and how it differs from procedural programming.

📚 What is OOP?

Object-Oriented Programming models software around objects that combine state (data) and behavior (methods). Instead of writing one long flow of steps, you design collaborating entities (e.g., User, Cart, PaymentGateway) that map to business concepts.

💡 Real-World Analogy

A car is an object: it has data (speed, fuel, engine state) and behavior (start, brake, accelerate). OOP says software should be built similarly using domain entities.

📄 OOP vs Procedural (mental model)

Procedural:
- Program organized around functions
- Data passed around between functions
- Easier to start, harder to scale

OOP:
- Program organized around objects/classes
- Data + behavior live together
- Better for large, evolving systems

▶ Why companies use OOP

  • Large codebases stay modular and easier to maintain.
  • Teams can work on separate classes/modules independently.
  • Reusability through inheritance/composition reduces duplication.
  • Encapsulation protects invariants and reduces bug surface.

🌟 Interview Insights

  • Define OOP in one sentence: “Modeling software as interacting objects with state and behavior.”
  • Always compare with procedural style when asked “Why OOP?”
  • Mention trade-offs: OOP helps scale design, but over-engineering is a risk for tiny scripts.

▶ Theory Deep Dive

Object-oriented design should be explained through abstraction boundaries, coupling/cohesion, and maintainability trade-offs.
