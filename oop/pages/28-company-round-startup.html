Startup-Style OOP Round

Typical pattern: build quickly, justify decisions, and show pragmatic evolution path.

📚 What they usually test

  • Can you ship a workable design fast?
  • Can you identify what to simplify today and what to abstract later?
  • Can you avoid obvious technical debt traps?
  • Can you discuss scaling path if usage grows 10x?

📄 “Pragmatic first” approach

Phase 1: Minimal cohesive classes, no over-abstraction
Phase 2: Add interfaces at external boundaries
Phase 3: Extract strategies where variability appears
Phase 4: Add observability, retries, and resilience layers

▶ Common prompts

  1. Design a lightweight order service with pluggable payments.
  2. Design basic task scheduler with future extensibility.
  3. Build API client SDK model with retry + fallback hooks.
  4. Take rough code and propose 2-step refactor roadmap.

🌟 Scoring booster

  • Say: “I’ll start simple and define extension seams where change is likely.”
  • Highlight observability and failure handling early.
  • Give one migration plan from MVP design to robust architecture.

▶ Theory Deep Dive

Object-oriented design should be explained through abstraction boundaries, coupling/cohesion, and maintainability trade-offs.
